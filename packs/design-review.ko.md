이 팩이 주입됐다는 것은 이번 작업이 UI 산출물 생성·수정(.html/.tsx/.vue/.svelte/.css 또는 domain=UI)으로 이미 판정됐다는 뜻이다. UI를 만지기 전 **규칙·토큰 SSOT `~/.claude/DESIGN-OPS.md`를 읽어라** — 아래는 그 규칙의 요약과 주의 신호일 뿐, 판정의 근거 원문은 그 파일이다. 프로젝트 `design/tokens.*`(§2)가 있으면 그 토큰을 상속하고, 없으면 부트스트랩(§2.0)이 선행돼야 한다.

## 1. 전역 하한 — 어길 때의 주의 신호 (DESIGN-OPS §1 참조)

아래는 모든 한국어 UI의 불변 하한이다. 규칙 원문·수치는 §1이 SSOT이며, 여기서는 **위반을 알아채는 신호**만 든다:

- **폰트(§1.1)**: 본문 Pretendard·코드 고정폭. 신호 — 임의 웹폰트를 새로 로드하고 있으면 위반.
- **타이포 상한(§1.2)**: 히어로·최대 제목 ≤ 2.7rem(프로젝트 토큰이 재정의 가능). 크기가 애매하면 **작은 쪽**으로, 대형 장식 타이포는 기본 기각. "폰트가 너무 커서 촌스럽다"는 이 계열에서 2회 기각된 실제 피드백이다 — 키우고 싶은 충동을 의심하라.
- **줄바꿈(§1.3)**: 전역 `word-break: keep-all` + 제목 `text-wrap: balance`. 신호 — **단어 중간에서 줄이 바뀌면** 절대 위반.
- **AI냄새 제거(§1.4)**: 신호 — 히어로당 그라데이션 2개 이상, 장식용 이모지, 보라(#7C3AED류) 단독 브랜드색, 제네릭 스톡 카피·히어로. 단 이건 최종 자동 판정 대상이 아니다 — 1차만 거르고 최종은 **사람(개입③)** 몫이다(본 팩 §2-c).
- **색(§1.5)·간격(§1.6)·다크모드(§1.8)**: 신호 — CSS 색 속성의 raw hex, 간격의 raw px, 하드코딩된 다크모드 색. **토큰만 참조**하라. 예외경계(토큰 원본 파일·`0`·헤어라인·차트색·allowlist 등)는 DESIGN-OPS §2가 규정한다.
- **대비·접근성(§1.7)**: WCAG AA(본문 4.5:1, 큰 텍스트 3:1) 이상. 신호 — 대비 하나로 접근성을 대표하지 마라. 키보드 탐색·`focus-visible`·`prefers-reduced-motion`·터치 타깃 ≥44px·시맨틱 마크업까지 본다.
- **모션(§1.9)**: 200~300ms ease 기본. 신호 — 과잉 애니메이션, `prefers-reduced-motion` 무시.
- **렌더 검증(§1.10)**: UI 산출물은 실제 구동 → 관측 → 수정 → 재구동. **정적 검사는 관측이 아니다**(본 팩 §3).

## 2. 판정 방식은 항목마다 다르다 — 정적으로 "확인함"이라 못 하는 게 대부분 (DESIGN-OPS §4 참조)

§4 체크리스트 8항은 판정 방식이 세 갈래다. **이 구분이 이 팩의 핵심**이다 — 대부분은 파일만 봐서는 통과를 주장할 수 없다:

- **(a) 정적 — 린트/AST로 판정**: ① 토큰 하드코딩 0(§4-1) · ⑧ 컴포넌트 상태 *명세* 존재 · ⑥ 다크모드 *토큰 참조* 여부. 파일만 봐도 검사된다(`fable_lite check --design`이 잡는다 — 구현은 Phase 2).
- **(b) 렌더 후 관측으로만 판정**: ② 제목 ≤ 상한(computed style — `clamp()`·상속·반응형은 정적 CSS로 못 잡는다) · ③ 단어 중간 줄바꿈 0(라인박스) · ④ WCAG AA + a11y(axe, 상태별) · ⑥ 다크모드 *실제 대비* · ⑦ 반응형 오버플로(뷰포트별) · ⑧ 상태별 렌더(hover·focus·disabled·loading·empty·error). **이 항목들은 렌더해서 눈으로 봐야만 판정된다.**
- **(c) 사람 눈 필수 — 자동 게이트 아님**: ⑤ AI냄새(§1.4). 임계·기준 이미지가 없고 심사 AI는 자기통과 편향이 있어 이진으로 자동 판정하지 않는다. §1.4 관찰 기준으로 1차만 거르고, 최종은 사람(개입③)이 "좋다 / 이건 다르다"로 판정한다.

## 3. 시안 "완료"의 증거는 렌더 검증 tool 실행이다 — 말이 아니라

(b) 항목들은 함께 주입된 `verification-grounding` 팩의 **관측(OBSERVE) 단계**에서 확인한다. 시안을 다 만들었으면 파일만 쓰고 "규칙 지켰습니다"라고 끝내지 마라:

- 렌더 검증 tool(playwright / chrome-devtools)로 **모바일 375·데스크톱 1280 뷰포트, light/dark 테마**에서 실제로 렌더하고 스크린샷을 관측하라.
- computed 타이포가 상한 이내인지, 단어 중간 줄바꿈이 없는지, 대비가 AA 이상인지, 상태별(hover/focus/disabled/loading/empty/error) 렌더가 온전한지 **눈으로 확인**하라.
- "토큰만 썼다 · 대비 맞췄다"고 **말로 선언하지 마라**. 정적 검사와 자기선언은 관측이 아니다. 관측 없이 확신을 흉내 내는 표현("맞을 것이다", "AA일 것이다")은 금지다 — 확인 안 했으면 안 했다고 써라.

이 팩의 목적은 규칙을 반복하는 데 있지 않다 — **DESIGN-OPS를 실제로 지켰음을 렌더로 보이게** 하는 데 있다.

### 게이트 계약 (디자인 Stop 게이트 · opt-in)

디자인 게이트는 프로젝트 opt-in이다(기본 OFF — 환경변수 `FABLE_LITE_DESIGN_GATE=1` 또는 `design/gate.config`의 `enabled: true`로 켠다). 켜진 상태에서 UI 파일을 변경한 턴(`ledger.design_required` AND `design_touched`)이면 Stop 훅이 두 가지를 판정한다:
- ① `fable_lite check --design`(A층 정적 린트) 미통과 → 차단("디자인 규칙 위반: `<파일:라인>`. DESIGN-OPS §N 참조").
- ② 렌더 검증 tool(B층) 호출 기록이 없음 → 차단("UI 변경인데 렌더 관측 없음. 스크린샷으로 확인 후 완료").

연속 차단은 최대 2회, 이후 통과(fail-open). Bash·조사·질문 전용 턴은 차단하지 않는다. 통과 시 검사 결과·스크린샷 경로가 완료 증거로 ledger에 기록된다.
단 A층 정적 린트(`design_lint`)와 이 Stop 디자인 차단은 쇼미더워크 재활성화 이후 작동하는 Phase 2 항목이다 — 그전까지 이 팩은 규율 주입(소프트)으로만 작동하니, 게이트가 안 잡아도 규칙 준수는 네 몫이다.
이 게이트는 네 "주장 문장"이 아니라 **tool 호출 기록**만 본다 — 말로 둘러대는 것은 통하지 않는다. 본 팩 §1~§3을 스스로 지키면 걸릴 일이 없다.
