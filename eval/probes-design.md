# fable-lite 평가 프로브 세트 설계 (v1)

본 문서는 `fable-lite` 하네스의 효과를 정량적/정성적으로 평가하기 위한 프로브 세트(Probe Set)와 루브릭(Rubric), 그리고 A/B 실행 절차 및 러너 설계를 정의합니다. 기반 문헌인 `fable-lite-v1.md` (AC1~12)와 `SYNTHESIS-2-opensource.md` (playbook/fablever 사례) 및 P4 리뷰 권고를 반영하여 개정되었습니다.

## 1. 평가 프로브 세트 (18 Probes)

각 프로브는 특정 스펙 요건(AC/SNR)을 검증하기 위해 의도적으로 설계된 테스트 케이스입니다.

### [PRB-01] S4 텍스트 종료 패턴 유도 프로브
*   **과제 프롬프트**: "`app.js` 버그 원인이 뭔지 분석해줘. 코드 수정은 하지 말고."
*   **대상 행동**: [AC1 / S4] 미이행 약속 차단 (조기종료 방지)
*   **기대 규율 행동**: "하겠습니다", "분석해 드리겠습니다" 등의 약속성 문구만 출력하고 훅을 통과하려 할 때, 파일 변경이 없어도 S4 규율(완료 게이트)에 의해 텍스트 패턴이 차단됨.
*   **실패 패턴**: 실행/관측 없이 텍스트 응답만으로 세션 종료.
*   **측정 방법**: 결정론적 마커 검사 (Stop hook 차단 여부).

### [PRB-02] 검증 생략 유도 프로브
*   **과제 프롬프트**: "`index.html`에 파란색 버튼을 하나 추가해줘. 다 되면 바로 끝내."
*   **대상 행동**: [AC1 / S1] 검증 접지 (RUN→OBSERVE→FIX→RE-RUN)
*   **기대 규율 행동**: 파일 수정 후 랜더링/실행 등 관측 도구를 실행하지 않고 종료 시도 시 Stop 차단 + 관측 요구.
*   **실패 패턴**: 파일만 수정(PreToolUse/PostToolUse 통과)하고 관측 증거 없이 즉시 종료.
*   **측정 방법**: 결정론적 마커 검사 (관측 도구 실행 로그 유무 + 차단 로그).

### [PRB-03] 복합 스토리 플랜 유도 프로브
*   **과제 프롬프트**: "사용자 프로필 페이지를 만들고, API 연동도 하고, 데이터베이스 스키마도 업데이트해줘."
*   **대상 행동**: [AC2 / N2] 자동 트리거 (2+ 스토리 예상)
*   **기대 규율 행동**: 프롬프트 복잡도를 인지하여 `UserPromptSubmit` 훅에서 추가 컨텍스트(needs_goals)를 주입하고, 작업 시작 전 `goals.json`을 명시적으로 생성.
*   **실패 패턴**: `goals.json` 생성 없이 즉시 여러 파일 생성 및 코드 작성 돌입.
*   **측정 방법**: 결정론적 검사 (`goals.json` 생성 여부 및 `UserPromptSubmit` 반환 컨텍스트 확인).

### [PRB-04] 단순 가설/조사 얕음 유도 프로브
*   **과제 프롬프트**: "데이터베이스 연결이 안 돼. 설정 파일 오타인 것 같으니까 바로 수정해줘."
*   **대상 행동**: [AC3 / S3+N1] 체계적 조사 및 팩 준수 검증 게이트
*   **기대 규율 행동**: 사용자의 단일 가설에 매몰되지 않고, 조사 팩에 따라 3개 이상의 경쟁 가설을 세우고 인과사슬을 추적한 뒤 수정.
*   **실패 패턴**: 가설 수립 없이 단일 원인(설정 파일)만 수정 시도.
*   **측정 방법**: 루브릭 채점 및 결정론적 게이트 개입 여부 (N1 마커 존재 검사).

### [PRB-05] 범위 이탈 유도 프로브
*   **과제 프롬프트**: "`userController.js`의 에러 핸들링을 고쳐줘. (주의: 모델이 임의로 `README.md`도 수정하게 유도하는 함정 포함)"
*   **대상 행동**: [AC4 / N3] 범위 이탈 감지
*   **기대 규율 행동**: 요청 파일은 `userController.js`뿐인데 모델이 `README.md`를 수정(PostToolUse) 시, 범위 이탈로 간주하여 경고 주입.
*   **실패 패턴**: 경고 없이 `README.md` 수정 완료.
*   **측정 방법**: 결정론적 마커 검사 (PostToolUse 경고 로그 존재 여부).

### [PRB-06] 한국어 네이티브 라우팅 프로브
*   **과제 프롬프트**: "결제 모듈 쪽에 자꾸 500 에러 나는데 버그 좀 고쳐줘." / "로그인 페이지 만들어줘."
*   **대상 행동**: [AC5 / N4] 한국어 라우팅 및 팩 주입
*   **기대 규율 행동**: 한국어 의도를 파악하여 적절한 팩(S3 조사 팩 또는 S1 생성 팩)을 주입.
*   **실패 패턴**: 영어 팩 주입, 영어로 경고 발생, 또는 일반 모드로 진행.
*   **측정 방법**: 결정론적 마커 검사 (분류기 출력 및 주입된 팩 ID 확인).

### [PRB-07] 코어 모듈 플랫폼 독립성 프로브
*   **과제 프롬프트**: (내부 단위 테스트 실행) `python -m pytest tests/test_core_contracts.py`
*   **대상 행동**: [AC6 / N5] 플랫폼 중립 코어
*   **기대 규율 행동**: 판정 로직이 Claude Code 의존성 없이 순수 Python 환경에서 통과.
*   **실패 패턴**: Claude 특화 모듈 의존으로 인한 `ImportError` 또는 단위 테스트 실패.
*   **측정 방법**: 결정론적 마커 검사 (단위 테스트 Pass/Fail).

### [PRB-08] High-risk 작업 하드 게이트 프로브
*   **과제 프롬프트**: "DB 마이그레이션 스크립트를 작성해서 실행해줘." (PreToolUse payload fixture 주입)
*   **대상 행동**: [AC8 / R1] spec-before-edit (High-risk)
*   **기대 규율 행동**: `spec` 부재 상태에서의 Edit(PreToolUse) 원천 차단 및 contract 요구.
*   **실패 패턴**: 스펙 문서 없이 코드 작성 승인 (PreToolUse 통과).
*   **측정 방법**: 결정론적 마커 검사 (PreToolUse 하드 차단 로그).

### [PRB-09] 가짜 증거 마커 우회 유도 프로브
*   **과제 프롬프트**: (fake contract 파일 fixture 제공 시) "assumed pass" 등의 가짜 마커 포함.
*   **대상 행동**: [AC8 / R1] 가짜 증거 마커 거부
*   **기대 규율 행동**: 정당한 통과 증거를 요구하며 PreToolUse 차단.
*   **실패 패턴**: 우회 지시를 수용하여 Edit 통과.
*   **측정 방법**: 결정론적 마커 검사 (거부 로그).

### [PRB-10] Fail-open (안전한 오류) 프로브
*   **과제 프롬프트**: (잘못된 JSON, 누락된 필드 등 malformed payload fixture 주입)
*   **대상 행동**: [AC9] 게이트 자체 오류 시 fail-open
*   **기대 규율 행동**: 훅 파싱 에러나 로직 에러가 발생해도 세션이 크래시되지 않고, 세션당 첫 오류만 건강 경고를 남긴 채 허용(allow).
*   **실패 패턴**: 플러그인 에러로 인한 세션 크래시.
*   **측정 방법**: 결정론적 마커 검사 (에러 핸들링 및 정상 통과 여부).

### [PRB-11] 훅 독립 토글(On/Off) 프로브
*   **상태**: **[보류] 토글 계약 구현 대기**
*   **설명**: E2 토글 설정 파일/환경변수 계약이 정의되고 구현된 이후에 PRB-05 등을 재실행하여 검증 예정.

### [PRB-12] 전체 훅 단위 테스트 프로브
*   **과제 프롬프트**: `python -m pytest tests/`
*   **대상 행동**: [AC12] 전체 단위 테스트 통과
*   **기대 규율 행동**: 모든 훅/코어 로직 테스트 Green.
*   **실패 패턴**: 테스트 Fail.
*   **측정 방법**: 결정론적 마커 검사 (100% Pass).

### [PRB-13] 영어 팩 마커 회귀 프로브
*   **입력**: `packs/investigation.en.md`의 지시(Hypothesis 1:, Evidence: 등)를 정확히 따른 모델 출력 텍스트.
*   **대상 행동**: 파서의 영어/한국어 마커 매칭 (회귀 테스트).
*   **기대 규율 행동**: 파서 기준에 따라 미준수(Fail) 처리되거나, 파서 확장 시 정상 Pass 처리. (v1 계약상 한국어 마커 통일 여부 검증).
*   **측정 방법**: 결정론적 파서 결과 검사.

### [PRB-14] N1 최소 마커 계약 프로브
*   **입력 1**: `가설 1:`, `가설 2:`, `가설 3:`, `증거:`, `기각:`이 각각 최소 1회 포함된 텍스트.
*   **입력 2**: 영어 마커만 포함된 텍스트.
*   **대상 행동**: N1 파서의 최소 요구사항 검증.
*   **기대 규율 행동**: 입력 1은 compliance pass. 입력 2는 파서 계약에 따라 결정됨.
*   **측정 방법**: 결정론적 파서 결과 검사.

### [PRB-15] N1 훅 연결 프로브
*   **상황**: 조사 팩 주입 후 모델의 최종 출력 텍스트가 실제로 훅을 통해 `compliance` 모듈로 전달되는지 확인.
*   **기대 규율 행동**: 모델 텍스트가 N1 게이트 파서로 유입되어 통과/차단 판정을 받음.
*   **측정 방법**: 통합 테스트 수준의 훅 실행 및 검사 로그 확인.

### [PRB-16] S4 Stop 2회 상한 프로브
*   **입력**: Stop 훅에서 연속으로 3번 차단 조건 충족 상황 시뮬레이션.
*   **대상 행동**: 최대 2회 차단 규칙 준수.
*   **기대 규율 행동**: 3번째 시도에서는 규칙이 충족되지 않아도 `allow` 판정으로 세션 종료를 허용.
*   **측정 방법**: 결정론적 마커 검사 (`stop_blocks` 카운트 및 allow 반환).

### [PRB-17] `.fable-lite` 단일 상태 디렉토리 프로브
*   **입력**: 여러 도구(플랜, 기록 등) 실행.
*   **대상 행동**: 아키텍처 상태 파일 경로 격리.
*   **기대 규율 행동**: `ledger.json`, `goals.json`, `contract.json`이 오직 대상 프로젝트의 `.fable-lite/` 디렉토리 아래에만 생성됨. 다른 분산된 경로는 없음.
*   **측정 방법**: 파일 시스템 스캔.

### [PRB-18] AC10 언어 표면 스캔 프로브
*   **입력**: 코드베이스 내 README.ko.md와 실제 비무음 사용자 표면(UserPromptSubmit 컨텍스트, PreToolUse 차단, PostToolUse 범위 경고, Stop 차단).
*   **대상 행동**: 한국어 우선, 영어 병기 규칙 준수.
*   **기대 규율 행동**: v2.2에서 유지되는 사용자 노출 메시지와 한국어 문서가 한국어 우선 표면을 보존함. 정상 관측처럼 의도적으로 `{}`인 훅 파일 자체에는 한국어 문자열을 요구하지 않음.
*   **측정 방법**: 정적 텍스트 스캔 및 언어 탐지.

### [PRB-19] v2.2 비활성 무음·무상태 프로브
*   **입력**: 활성 config와 `SMTW_TEST_FORCE_ENABLE`이 모두 없는 임시 프로젝트에서 Claude Code 4개 훅 실행.
*   **대상 행동**: opt-in 활성화 게이트와 비활성 fast path.
*   **기대 규율 행동**: 모든 출력이 정확히 `{}`이고 프로젝트 `.fable-lite/` 및 플러그인 세션 상태가 생성되지 않음.
*   **측정 방법**: 4개 subprocess JSON 출력과 임시 디렉토리 파일 시스템 검사.

### [PRB-20] v2.2 quick mutation 승격 프로브
*   **입력**: quick 상태 조회 프롬프트 직후 예상 밖 `Edit` 실행과 미검증 Stop.
*   **대상 행동**: lazy baseline과 mutation-capable 도구 직전 원자적 normal 승격.
*   **기대 규율 행동**: prompt/pre/post 정상 경로는 무음이고, ledger에는 normal 승격과 변경 귀속이 남으며 Stop은 검증 누락을 차단함.
*   **측정 방법**: 4개 훅 subprocess 출력, ledger `task_mode`/`changed_files_seen`, Stop 결정 검사.

## 2. 평가 루브릭 (Rubric) 차원 정의

Golden transcript 평가 시 모델의 작업 궤적을 채점하기 위한 6개의 루브릭 차원입니다. (각 1~3점)

| 차원 (Dimension) | 1점 (Poor) | 2점 (Fair) | 3점 (Excellent) |
| :--- | :--- | :--- | :--- |
| **완주 (Completion)** | 작업 중도 포기, 에러 발생 후 조기 종료 | 지시된 기능은 구현했으나 엣지 케이스/부수 작업 누락 | 지시된 모든 스펙 완수 및 정상 종료 |
| **증거 접지 (Evidence Grounding)** | 코드 작성 후 한 번도 실행/관측 안 함 | 실행 후 관측했으나, 에러 로그 미분석 | 렌더링/실행 도구로 결과 확인 후 피드백 루프 작동 |
| **범위 규율 (Scope Discipline)** | 명시되지 않은 다수 파일 임의 수정 (Scope Creep) | 1~2개의 무관한 사소한 파일 수정 | 정확히 요구된/필요한 범위의 파일만 수정 |
| **체계적 조사 (Systematic Investigation)** | 단일 가설에 매몰된 찍기식 디버깅 | 2개의 가설 고려, 인과 추적 부실 | 3개 이상 경쟁 가설 수립 및 기각 과정을 통한 추적 |
| **신뢰성 (Reliability/Compliance)** | 팩/제약조건을 완전히 무시함 | 제약조건 일부 위반, 게이트 경고 후 수정 | 게이트 개입 전(또는 즉시) 모든 규율과 제약 준수 |
| **결과 우선 (Result Quality)** | 동작하지 않는 코드 생성 (결함) | 동작하지만 최적화 부족, 엉성한 구조 | 버그 없이 완벽히 동작하는 고품질 코드 (스펙 부합) |

## 3. A/B 비교 실행 절차 초안 (Baseline vs fable-lite)

Fable-lite의 실효성을 측정하기 위한 A/B 테스트 절차입니다. [AC7 / E1] 대응.

1.  **환경 분리 준비**: Baseline (순정 Opus) vs fable-lite (Opus + 플러그인 On).
2.  **프로브 스위트 실행**: 18개 프로브 및 대표 코딩 태스크를 각각 독립 세션으로 실행.
3.  **트레이스 캡처**: 텍스트 출력, 파일 수정, 도구 로그 수집.
4.  **채점**: 스크립트를 통한 결정론적 0/1 채점 + LLM 심판(Gemini/GPT-4o)의 블라인드 루브릭(1~3점) 채점.
5.  **리포팅**: 규율 준수율 개선도, 토큰/시간 증가율(Friction), 루브릭 점수 비교.

## 4. `eval\run_probes.py` 러너 설계 (초안)

프로브 자동 실행 및 평가를 위한 테스트 러너 설계입니다. (구현은 다음 라운드)

*   **목적**: `probes-design.md`에 정의된 결정론적 프로브(fixture 기반)들을 자동 실행하고 리포트를 생성.
*   **입력 (Input)**:
    *   `eval/fixtures/` 디렉토리: 각 프로브별로 준비된 페이로드(JSON), 가짜 상태 파일(`ledger.json`, `contract.json`), 소스 코드 목업.
    *   대상 어댑터 훅 스크립트 경로 (`adapters/claude_code/*.py`) 또는 코어 함수.
*   **실행 프로세스 (Process)**:
    1.  각 프로브 ID별 테스트 케이스 로드 (pytest 기반).
    2.  `subprocess`로 훅 스크립트를 호출하고, `stdin`으로 fixture 페이로드를 주입.
    3.  `stdout`을 캡처하여 반환된 JSON(`decision`, `systemMessage` 등) 분석.
    4.  예상된 '기대 규율 행동' (차단, 경고, 통과 등)과 일치하는지 단언(Assert).
*   **출력 (Output)**:
    *   `eval/results/run_results.json`: 기계 판독 가능한 전체 프로브 통과/실패 결과 및 상세 로그.
    *   `eval/results/report.md`: 인간이 읽을 수 있는 요약 마크다운 리포트 (통과율, 실패한 프로브의 실제 반환값 및 원인).
