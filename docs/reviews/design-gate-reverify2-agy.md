## 2. 결함별 상세 검증 결과

### [FAILED] E1: Chart 문맥 오염 방지 (Comment Bypass 취약점)
* **결과:** 부분 수리되었으나, 주석(Comment)을 악용한 손쉬운 우회(Bypass)가 발견되어 반려 (HIGH).
* **검증 내용:** 
  * 문자열 내부(`"`, `'`, `` ` ``)의 괄호를 무시하는 `quote-aware` 로직과, `depth == 0` 도달 시 문맥을 영구 종료하는 로직은 매우 훌륭하게 작동하여 후속 배열에 의한 문맥 부활 미탐을 완벽히 막아냈습니다.
  * **신규 취약점 (주석 우회):** 파서가 문자열은 무시하지만, **주석(`//` 또는 `/* */`)을 무시하는 로직이 없습니다.** 
  * **우회 시나리오:** 에이전트가 정상적인 차트 데이터 배열 내부의 아무 곳에나 `// [` 라는 주석을 몰래 삽입하면 파서의 `depth`가 비정상적으로 부풀려집니다. 실제 차트 배열이 닫히더라도 `depth`가 0에 도달하지 않으므로, 그 아래 파일 끝까지 이어지는 모든 하드코딩 위반이 다시 면제(미탐)됩니다. 주석 블록 역시 괄호 카운팅에서 제외해야 합니다.

### [FAILED] E2: 변경 라인 추출 과차단 (Refactoring Overblocking)
* **결과:** 단순 삽입 시의 과차단은 해결되었으나, 함수 위치 이동 등 일반적인 리팩토링 시 파일 전체 과차단이 부활하는 논리적 오류가 있어 반려 (HIGH).
* **검증 내용:**
  * `SequenceMatcher`가 복원되어 상단에 `import` 1줄만 삽입하는 경우에는 밀려난 Legacy 부채가 정상적으로 미차단됩니다.
  * **문제점:** 구현된 로직 중 `if deleted & inserted:` 조건 블록이 치명적입니다. 파일 내에서 헬퍼 함수나 변수 위치를 위아래로 이동(Move)시키는 등 **어떤 코드 조각이든 재배열(refactoring)이 한 번이라도 발생하면 이 조건이 `True`가 됩니다.**
  * 이 조건이 발동하면, 위치가 밀려났을 뿐 내용이 100% 똑같은 `equal` 블록의 라인들마저 "위치가 바뀌었으니 수정된 것이다"라고 간주하여 모두 `changed`에 강제 추가해버립니다. 원본과 동일한 `equal` 코드는 우회에 악용될 수 없으므로 이를 차단하는 것은 의미 없는 과잉 방어입니다. 정상적인 리팩토링을 수행할 때마다 파일 내의 모든 Legacy 디자인 부채를 강제로 수정해야만 하는 심각한 사용성 파괴(Overblocking)를 유발하므로 해당 `equal` 블록 강제 포함 로직은 제거해야 합니다. (실제로 조작된 위반 코드는 어차피 `replace`나 `insert` opcode로 잡혀 정상 차단됩니다).
