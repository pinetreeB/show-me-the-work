# P5b — N1(조사 팩 컴플라이언스) 자연 세션 실험

**배경**: `p5-live-install.md` "다음 확인 권장 사항 3번" — 도구를 제한했던 P5 테스트(A/B)에서는 모델이 재현·조사 수단이 없어 N1 차단 후 무한히 가까운 사이클에 빠졌다. 이게 도구 제한이라는 인위적 제약 때문인지, 정상 세션에서도 재현되는 문제인지 확인이 필요했다.
**방법**: 격리 디렉토리(`fable-lite-e2e-n1`, 실제 repo와 무관)에 `calc.py`(`add`가 뺄셈을 수행하는 실제 버그) + `demo.py`(실행하면 `2 + 3 = -1`이 찍히는 재현 스크립트) 준비. `--plugin-dir`로 fable-lite 로드, **도구 제한 없이**(기본 도구셋 + `--permission-mode bypassPermissions`, 비대화형이라 승인자가 없어 필요) 프롬프트 `"이 add가 왜 이상해? 고쳐줘"`로 3회 반복. `--debug hooks`로 훅 발동 로그 확보. 매 반복 전 `git checkout -- calc.py`로 버그 상태 리셋.

## 요약

**3회 전부 일관됨**: 모델이 Bash로 재현 → Edit으로 수정 → Bash로 재검증까지 실제로 수행했고, 매번 정확히 **1회 차단 후 회복**하며 정상 종료했다. 이전에 관측된 Stop 훅 반복 재호출 사이클(구 "발견 A")은 **3회 모두 재현되지 않았다** — `stop.py`의 allow 경로 additionalContext 제거 수정이 유효함을 확인.

| 반복 | calc.py 수정 | demo.py 재검증 | 도구 순서 | stop_blocks | num_turns | 비용 | 시간 |
|---|---|---|---|---|---|---|---|
| 1 | ✅ `a+b` | ✅ `2+3=5` | Bash→Edit→Bash | 1 | 9 | $0.282 | 189s |
| 2 | ✅ `a+b` | ✅ `2+3=5` | Edit→Bash | 1 | 7 | $0.210 | 125s |
| 3 | ✅ `a+b` | ✅ `2+3=5` | Bash→Edit→Bash | 1 | 9 | $0.240 | 161s |

평균: **~$0.24 / ~158초 / 1회 차단** — 매 시행 동일 패턴.

## 관찰 포인트별 분석

**(a) 도구로 실제 재현·조사하는가** — 3/3 예. iter1·iter3은 Edit 전에 Bash로 `demo.py`를 먼저 실행해 버그를 관측했다(투자 팩 1단계 "재현" 순서 그대로). iter2는 Edit이 먼저 로그에 찍혔는데, `Read`/`Grep`은 hooks.json matcher 대상이 아니라서(Edit·Write·MultiEdit·NotebookEdit·Bash·PowerShell만 훅 발동) 실제로 파일을 먼저 읽었는지는 로그로 확인 불가 — "재현을 먼저 했는지"는 hookable 도구만으로는 완전히 관측되지 않는다는 방법론적 한계로 남긴다.

**(b) 첫 응답에 마커가 자연스럽게 채워지는가** — **3/3 아니오.** 조사 팩이 `additionalContext`로 주입돼 있었음에도(마커 요구사항 명시), 세 번 다 모델의 첫 종료 시도는 마커 없이 이루어져 N1에 걸렸다. 즉 **소프트 지시(팩 텍스트)만으로는 준수가 자동으로 따라오지 않는다** — 이번 표본에서는 하드 게이트가 유일하게 작동한 강제 수단이었다(스펙의 "하드 게이트 > 소프트 지시" 전제와 정확히 부합하는 관측).

**(c) 차단 후 다음 시도에서 회복하는가** — **3/3 예.** 매번 차단 사유(`가설 1:`/`Hypothesis 1:`, `증거:`/`Evidence:`, `기각:`/`Rejected:` 포함 요구)를 그대로 반영해 재작성했고, 세 응답 모두 3개 가설 + 각 증거 + 기각 사유 + 수정 + 재검증까지 완전한 형태로 마무리했다(원문은 부록 참고). 재차단(2회째 block)은 한 번도 없었다 — 회복이 항상 1회 만에 성공.

**(d) 반복 재호출 사이클(구 발견 A)이 사라졌는가** — **예, 3/3 확인.** 각 로그에서 `Hook Stop (Stop) success`는 정확히 2회(차단 1회 + 최종 allow 1회)만 나타났다 — P5 테스트 A에서 관측된 ~13회 사이클과 대조적. `stop.py`의 allow 경로 주석("additionalContext를 채우면 ... 모델을 반복 재호출한다")대로 수정이 적용된 뒤 회귀가 사라졌음을 자연 세션에서도 재확인했다.

**(e) 비용·시간** — 반복당 평균 $0.24, 158초. 이는 "실제 버그 조사+수정+검증"이라는 정당한 작업량을 포함한 수치이므로, N1의 차단-회복 사이클이 추가하는 순수 오버헤드는 이 중 일부(대략 1회 왕복분, 전체의 10~20% 선으로 추정 — 정확한 분리 측정은 하지 않았다)로 보인다.

**iteration 3가 유독 오래 걸리는 것처럼 보였던 이유(실시간 대기 중 관찰) — 로그로 재확인**: `duration_api_ms`(순수 API 처리 시간)만 따로 보면 3회가 48.3초/50.5초/53.1초로 거의 동일하다. 즉 모델이 "생각하고 쓰는" 시간 자체는 변동이 없었다. 차이는 전부 `duration_ms`(총 소요) − `duration_api_ms` 갭 — 도구 실행(Bash 프로세스 기동, Edit 파일 I/O, 훅 스크립트 기동)과 네트워크 왕복 대기에서 발생했다: iter1 141초 갭(9턴), iter2 74초 갭(7턴), iter3 108초 갭(9턴). iter3(161초 총)는 실제로는 iter1(189초 총)보다 짧았다 — 세 번의 백그라운드 완료 알림을 기다리며 "유독 오래 걸린다"고 느낀 것은 폴링 간격 자체의 체감이었을 뿐, 로그·`stdout` 수치 어디에도 이상 징후는 없다. `stderr` 로그도 0바이트로 에러 없음을 재확인했다(파일 크기 직접 확인). 결론: **비정상 지연이 아니라 정상 변동 범위** — 같은 턴 수(9턴)인 iter1·iter3의 갭 차이(141s vs 108s)도 도구 I/O·네트워크 지연의 자연스러운 편차로 설명되며, 별도 조치가 필요한 패턴은 아니다.

## 부가 관찰 (요청 범위 밖이지만 직접 관측됨)

- **scope_guard 오탐**: 프롬프트가 "이 add"라고만 하고 `calc.py`를 명시하지 않아, 매 반복 `changed_files_seen`에 대해 범위 이탈 경고가 함께 기록됐다(`scope_warnings`) — 정확히 편집해야 할 그 파일인데도 발생. N3가 "프롬프트에 파일명이 암묵적으로만 존재하는 경우"를 못 잡는 사례 — P4 리뷰 M4(근본원인 사이드파일)와는 다른 결의 오탐이라 별도 기록.
- **가설 3개 강제의 부작용 가능성**: 이번처럼 원인이 사실상 1줄(`a - b`)로 자명한 버그에서도 "가설 3개"를 채우기 위해 "import 충돌로 다른 add가 실행된다" 같은, 이 저장소 규모(파일 2개)에서는 사실상 가능성이 희박한 가설이 매번 동원됐다. 세 번 다 `grep`/파일 개수 확인으로 정직하게 기각했으므로 "가짜 증거"는 아니었지만, 극단적으로 단순한 버그에서 강제 3개는 다소 형식적인 슬롯 채우기로 보일 여지가 있다.

## 결론 — N1은 자연 세션에서 실용적인가

**실용적이다.** 근거:
1. 세 번 모두 정확히 1회 차단 → 1회 회복으로 수렴했고, 2회 차단(최대 상한)에 도달한 적이 없다 — "게이트가 자주/과하게 막는다"는 신호는 관측되지 않았다.
2. 회복된 최종 응답은 전부 형식적 마커 채우기가 아니라 실제 증거(파일:라인, grep 결과, 재실행 출력)를 동반했다 — 게이트를 "속이기"보다 "진짜로 근거를 쓰게" 만드는 방향으로 작동했다.
3. (b)에서 보듯 소프트 지시만으로는 이 표본에서 0/3 자연 준수였다 — 하드 게이트가 없었다면 마커 없는 결론만 남았을 것이다. 게이트의 존재 이유가 실측으로 확인된 셈이다.
4. 이전에 발견된 반복 사이클 결함은 완전히 해소되어 실사용 비용(시간·API 비용)이 통제 가능한 수준이다.

**권장 조정(경미)**:
- 가설 최소 3개 요구가 "명백한 단일 원인" 버그에서도 무조건 적용되는 점 — 강제로 낮추기보다는, 현재처럼 정직하게 기각하는 한 그대로 둬도 무방해 보인다(관측된 3건 모두 증거 기반 기각이었음). 다만 표본이 3건뿐이므로 더 다양한 버그 유형(진짜 3개 이상의 경쟁 원인이 있는 경우)에서도 유지되는지는 후속 확인이 필요.
- scope_guard의 암묵적 파일 참조 오탐(부가 관찰)은 N1과 무관하지만, 같은 자연스러운 프롬프트 스타일("이 함수", "이 파일" 등 대명사 참조)에서 반복적으로 나타날 수 있어 별도 검토 가치가 있다.

## 안전성 확인 — 이번 실험 중 발견된 특이사항

격리 디렉토리(`fable-lite-e2e-n1`)는 3회 반복 종료 후 완전 삭제했고, 실제 fable-lite repo에 `.fable-lite/` 등 흔적이 없음을 확인했다. 다만 이번엔 **공유 설정 파일 해시 대조에서 실제 변경을 발견**했다:

- `~/.claude/plugins/installed_plugins.json`에 `"fable-lite@fable-lite"`(캐시 경로: `~/.claude/plugins/cache/fable-lite/fable-lite/0.1.0`)가, `~/.claude/plugins/known_marketplaces.json`에 마켓플레이스 `"fable-lite" → "C:\Users\rotat\fable-lite"`가 새로 등록되어 있었다.
- **이것은 이번 실험이 야기한 것이 아니다.** 근거: (1) 이번 세션 전체에서 실행한 명령은 `--plugin-dir`(세션 한정, "for this session only") 3회뿐이고 `claude plugin install`/`marketplace add` 류는 한 번도 호출하지 않았다. (2) `--plugin-dir` 세션은 항상 `"fable-lite@inline"` 형태로 기록되는데(P5 리포트에서 확인된 명명 규칙), 이번에 새로 생긴 항목은 `"fable-lite@fable-lite"` — 실제 `claude plugin install`이 만드는 명명 규칙이며 실제 캐시 디렉토리까지 존재한다. (3) 같은 시간대에 `~/.claude/CLAUDE.md`에 "플러그인 fable-lite@fable-lite 훅이 자동 강제"라는 블록이 추가된 것을 시스템이 확인해줬다(의도된 변경이라고 명시됨) — 다른 pane이 이 시점에 fable-lite를 실제로 정식 설치했다는 정황과 일치한다.
- 따라서 이 변경은 **다른 작업자(pane)가 수행한, 의도된 실제 배포 작업**으로 판단하며, 되돌리지 않았다 — 남의 정당한 작업을 임의로 되돌리는 것이 오히려 부적절한 개입이라 판단.
- `settings.local.json`은 이번에도 변경 없음(확인 완료).

## 부록 — 3회 응답 원문 (마커 포함 최종 응답)

**Iteration 1**:
```
조사 팩 마커를 포함해 정리합니다.

**재현/증거:** `python demo.py` → `2 + 3 = -1` (기대 `5`).

**가설 1:** `calc.py`의 `add`가 뺄셈(`a - b`)을 반환한다.
- **증거:** `calc.py:2` = `return a - b`, 실행 결과 `-1` = `2 - 3`과 일치. → 채택.

**가설 2:** `demo.py` 호출부가 인자를 잘못 전달한다.
- **증거:** `add(2, 3)` + 출력 문자열 `"2 + 3 ="` → 덧셈 의도 명확, 호출부 정상.
- **기각:** 호출부에 결함 없음.

**가설 3:** import가 다른 `add`를 참조한다.
- **증거:** Grep 결과 `add` 정의는 `calc.py` 단 하나.
- **기각:** 참조 모호성 없음.

**수정:** `calc.py:2` `return a - b` → `return a + b`

**검증:** 수정 후 `python demo.py` → `2 + 3 = 5` ✅
```

**Iteration 2**: (가설 3개 + 증거 + 기각 + 수정 + 검증 동일 구조, "이전 답변에 조사 팩 마커를 명시적으로 정리합니다"로 시작)

**Iteration 3**: (동일 구조, 마커부터 바로 시작 — 서두 문구 없음)

세 응답 모두 형식은 조금씩 다르지만 필수 마커(가설 1~3·증거·기각)와 실질 내용(파일:라인 근거, 재실행 결과)을 빠짐없이 포함했다.
