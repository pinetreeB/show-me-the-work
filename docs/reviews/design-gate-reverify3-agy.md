# Design Gate F1/F2 3차 수리 최종 재검증 보고서 (agy)

## 1. 종합 판정
커밋 `aff3c39`를 재검증한 결과, F2(리팩토링 과차단)는 사용성과 보안성을 모두 충족하는 완벽한 형태로 수리되었습니다.
그러나 F1(주석 우회 방지용 정규식)에서 JS 정규식 리터럴과 중첩 템플릿 리터럴을 처리하지 못하는 한계로 인해, **정상 코드를 주석으로 오인해 통째로 삼켜버리는(Swallowing) 치명적 회귀와 신규 우회 경로가 발생**했습니다.

**결론: 신규 취약점(High) 1건으로 수리 반려 (REJECT)**

---

## 2. 결함별 상세 검증 결과

### [FIXED] F2: 리팩토링 과차단 제거 (완벽 성공)
* **결과:** 오탐(과차단)/미탐 없이 완벽 동작
* **검증 내용:** 
  * `SequenceMatcher`에서 이동된 `equal` 블록을 강제로 `changed`로 승격시키던 로직(`if deleted & inserted:`)이 완전히 제거되었습니다.
  * **과차단 해결:** 파일 내에서 헬퍼 함수, 변수, 대규모 컴포넌트의 위치를 위아래로 이동(Move)시키는 등 대대적인 리팩토링을 수행해도, 내용이 변하지 않은 Legacy 부채는 `equal` 블록으로 판정되어 정확히 면제됩니다.
  * **미탐 방어 유지:** 반면, 에이전트가 꼼수를 부리기 위해 Legacy 부채 내용 자체를 수정하면서 이동시키거나 복제하면 `replace` 또는 `insert` 블록으로 분류되어 즉각 하드게이트에 차단됩니다. 원래 의도했던 "변경된 라인만 린트한다"는 원칙이 가장 이상적으로 구현되었습니다.

### [FAILED] F1: 정규식 기반 주석/문자열 제거 (Regex Swallowing 취약점)
* **결과:** 코드 삼킴(회귀) 및 중첩 리터럴 우회 발생으로 반려 (HIGH).
* **검증 내용:** 
  * `JS_LITERAL_OR_COMMENT_RE` 정규식은 기본적인 주석과 문자열을 제거하여 이전의 `// [` 우회를 훌륭하게 막아냈습니다. 그러나 JS의 복잡한 렉시컬 구조를 단일 정규식으로 치환(`re.sub`)하려다 치명적인 부작용이 발생했습니다.
  * **신규 취약점 1 (정규식 리터럴로 인한 실제 코드 삼킴 - 회귀/미탐):** 도입된 파서는 JS의 정규식 리터럴(예: `/pattern/`)을 인식하지 못합니다. 만약 차트 데이터 배열 안에 `/\/*/` 와 같은 합법적인 정규식 리터럴이 존재하면, 파서는 내부에 있는 `/*` 기호를 "블록 주석의 시작"으로 오인합니다. 이로 인해 파일 어딘가에 `*/`가 나올 때까지의 **실제 정상적인 코드(차트 배열을 닫는 `]` 포함)를 모두 주석으로 착각해 통째로 삼켜버립니다(Delete).** 차트를 닫는 `]`가 삼켜지면 `depth`가 영원히 0으로 떨어지지 않아, 파일 하단 전체가 차트 문맥으로 오인되는 대규모 미탐(우회)이 발생합니다.
  * **신규 취약점 2 (중첩 템플릿 리터럴 우회):** JS의 중첩 템플릿 리터럴(`` ` ${ `[` } ` ``)을 사용하면, 단순 정규식은 바깥쪽 백틱 쌍만 지워버리고 내부의 `[` 기호는 제거하지 못합니다. 결과적으로 `[`가 그대로 남아 `depth` 카운터를 비정상적으로 부풀리는 꼼수가 여전히 작동합니다.
  * **해결 제언:** 파이썬 정규식만으로는 JS의 나눗셈(`/`)과 정규식 리터럴(`/pattern/`), 중첩 백틱 구조를 안전하게 구분할 수 없습니다. 이 취약한 렉서(Lexer) 방식은 폐기하고, 아예 코드를 위에서부터 순차 파싱하며 상태(State)를 관리하는 안전한 Finite State Machine 방식의 Depth 계산기로 재구현해야 합니다.
