# Design Gate 수정안 최종 재검증 보고서 (agy)

## 1. 종합 판정
수정된 커밋 `5344935`를 재검증한 결과, D1(Allowlist 세탁)과 D5(자가무력화) 등 치명적인 우회 경로는 코어 원천에서 완벽히 차단되었습니다. 정규식 확장(D3, D4) 역시 정상적으로 수행되었습니다.
그러나 **D2(Chart 문맥 오염)의 수리 로직에 중대한 논리적 결함이 있어 여전히 미탐이 발생**하며, 변경 라인 추출 로직이 잘못 축소되어 **전체 파일 과차단(Massive Overblocking)이라는 새로운 치명적 회귀 버그**가 발생했습니다.

**결론: 미해결 High 1건, 신규 회귀 High 1건으로 수리 반려 (REJECT)**

## 2. 기존 결함(D1~D5) 수리 검증 결과

### [FIXED] D1 (Allowlist 세탁) & D5 (자가무력화)
* **결과:** 완벽히 방어됨
* **근거:** `core/ledger_v1.py`에서 `design/gate.config` 파일의 변경 여부(`policy_changed`)를 추적하도록 수정되었습니다.
* 에이전트가 TOCTOU 세탁(위반추가 → allowlist 임시 허용 → check PASS → allowlist 몰래 삭제)을 시도하거나 `enabled: false`로 자가무력화를 시도해도, **설정 파일 변경 자체가 UI 변경과 동일하게 취급되어 즉각 `design_check_passed = False`로 무효화**하고 `design_last_change_seq`를 갱신합니다. 에이전트는 꼼수를 부려도 반드시 올바른 상태에서 재검사를 받아야 하므로 하드게이트를 절대 우회할 수 없습니다. 정상 사용자의 편집 역시 다시 검사만 돌리면 되므로 과차단이 아닙니다.

### [FAILED] D2 (Chart 문맥 오염 방지)
* **결과:** 수리 실패 (여전히 문맥 오염 발생, HIGH)
* **근거:** 수정된 `_chart_data_color()`는 배열/객체가 닫혔는지 판별하기 위해, 차트 시작점부터 현재 줄까지의 텍스트에서 괄호 개수(`preceding.count('[') > preceding.count(']')`)만 단순 비교합니다.
* **우회 시나리오:** 파일 상단에 `chartData`가 정상적으로 선언되고 온전히 닫히더라도, 그 아래 코드에서 무관한 새 배열/객체(예: `const b = [ ... ];`)가 열리면 `[` 개수가 누적 증가합니다. 결과적으로 `count('[') > count(']')` 조건이 다시 `True`가 되어버려, **새 배열 내부에 있는 무관한 Raw Color 하드코딩들이 모두 '열려있는 차트 색상'으로 오인되어 린트를 무사통과**합니다. 깊이(Depth)가 0으로 떨어지면 차트 문맥이 영구 종료되도록 제대로 파싱해야 합니다.

### [FIXED] D3 (정규식 확장) & D4 (1px 오탐 방지)
* **결과:** 오탐/미탐 없이 완벽 동작
* **근거:** `margin: "0 16px"` (shorthand), `box-shadow`, `--var`, `rgba()`, `[-13px]` 음수 Tailwind 등이 정규식에 모두 성공적으로 반영되었습니다. `abs(number) > 1` 로직 덕분에 DESIGN-OPS가 허용한 `0`과 `1px` 헤어라인은 안전하게 통과되며, `2px` 이상의 하드코딩만 정확히 차단됩니다. 차트 배열이 여러 줄로 작성되어도 오탐은 생기지 않았습니다.

## 3. 신규 회귀(Regression) 검증 결과

### [HIGH] 단순 줄 비교 도입으로 인한 전체 파일 과차단 (Massive Overblocking)
* **결과:** 심각한 과차단 회귀 발생
* **근거:** `core/design_lint.py`에서 기존 `SequenceMatcher` 로직을 제거하고, 단순히 인덱스 1:1 기반의 해시 비교(`digest != baseline_hashes[index]`)로 변경 라인을 추출하도록 수정되었습니다.
* **문제점:** 파일 맨 위(예: 라인 1)에 `import` 문 하나만 새로 추가하거나 지워도, 그 아래에 있는 1000줄의 코드가 모두 한 줄씩 밀려나면서 인덱스가 어긋납니다. 로직 상 인덱스가 어긋난 **모든 라인이 "방금 수정된 라인"으로 오인**됩니다. 
* 결과적으로 에이전트가 코드 상단에 한 줄만 추가해도 해당 파일 전체가 린트 대상이 되며, 파일 내에 숨어있던 모든 Legacy 디자인 부채(raw color, raw spacing 등)가 무더기로 적발되어 하드게이트가 블록됩니다. 이는 변경 라인만 검사하겠다는 "Dirty Line Baseline"의 근본 취지를 파괴하는 치명적 회귀입니다.
