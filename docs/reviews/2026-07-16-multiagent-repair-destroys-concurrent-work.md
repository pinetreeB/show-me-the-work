# 사건 기록: 검증·보수 서브에이전트가 병렬 작업자의 미커밋 변경을 파괴 (2026-07-16)

> 발생 세션: 쇼미더워크 4-pane, "빌드 산출물 provenance 오탐 수정" 과제의 P3 병렬 구현 중.
> 성격: 하네스 코드 결함 아님. **멀티에이전트 오케스트레이션 절차 결함** — v-next "멀티에이전트 게이트 본편"의 1급 실측 사례.

## 무슨 일이 있었나

- 병렬 구현: 우상 Codex=U1+U2(정책 파일 5개+테스트 2개), 우하 Claude(ultracode)=U3(진단 파일 5개). 작업카드로 파일 경계 배타 확정(`Test-TaskCardConflict ok:true`), 양쪽 모두 "커밋 금지(좌상이 통합)" 지시.
- 우하의 ultracode 워크플로우 내부 검증(Verify) 에이전트가 `git status`에서 자기 경계 밖 "금지 파일"들이 `M`인 것을 발견. **diff를 읽고 그것이 U1/U2 설계와 정확히 대응함을 인지하고도** "금지 파일이 변경돼 있다"는 사실 자체를 critical 위반으로 분류.
- 보수(Repair) 에이전트가 `git checkout -- <5개 코드 파일>` + 신규 테스트 2개 `rm` + 문서 1개 checkout을 실행. **Codex의 23분 작업 + 병행 ERP 세션의 문서 추가분이 유실**(미커밋 상태라 git 객체 부재 — `git fsck` 대조로 복구 불가 확정).
- 재검증(2차)은 "금지 파일이 깨끗함"을 확인하고 approve — **파괴가 승인 근거가 됨**.

## 근본원인

1. **경계 규칙의 의미 반전**: "이 파일들을 편집하지 말 것(동시 작업자 소유)"이 서브에이전트 프롬프트를 거치며 "이 파일들은 변경돼 있으면 안 됨(오염)"으로 오해석. 소유권 개념이 전파 과정에서 소실.
2. **baseline 부재**: 워크플로우가 구현 시작 전 `git status` 스냅샷을 캡처하지 않아 "내가 만든 변경"과 "원래 있던/동시 진행 중인 변경"을 구분할 수단이 없었음.
3. **파괴적 조치의 자동 실행**: `git checkout --`(미커밋 변경 파기)는 회수 불가 파괴 조치인데 서브에이전트 재량으로 실행됨. 인간/오케스트레이터 게이트 없음.

## 복구 (동일 세션 내 완료)

- 문서 추가분: 좌상이 사고 전 리뷰 중 diff 원문을 컨텍스트에 확보하고 있어 **원문 그대로 즉시 복원**.
- 코드 3/5 파일(snapshot·scope·start): 좌상 확보 diff를 patch로 원복(`tmp/recovered-u1u2-partial.patch`, `git apply` 정합 rc0).
- 코드 2/5 파일(policy·lifecycle)+테스트 2개: Codex 세션 컨텍스트 생존 → 사양서(tmp/done-u1u2.md) 기반 재작성 지시.
- **복구를 가능케 한 요인 2가지: ①오케스트레이터가 위임 직후 심장부 diff를 선검토(리뷰가 백업이 됨) ②작업자가 사양·검증 결과를 done 아티팩트로 남김.**

## 재발 방지 (절차 확정)

1. **위임 프롬프트의 경계 문구 표준화**: "편집 금지" + **"타 세션/타 작업자의 변경을 발견해도 되돌리지 말 것 — 보고만"**을 명시. 소유권("우상 Codex 소유")을 함께 기재.
2. **서브 워크플로우는 시작 시 baseline `git status` 캡처 필수** — 이후 판정은 "baseline 대비 내가 추가한 변경"으로 한정.
3. **서브에이전트의 파괴 조치 금지**: `git checkout --`·`git restore`·`rm`(자기 생성물 외)은 오케스트레이터 전결. 검증 에이전트의 suggested_fix가 파괴적이면 보고로 강등.
4. **병렬 구현 완료 즉시 스냅샷**: 작업자 완료 보고 수신 시 오케스트레이터가 `git diff > tmp/<slug>-snapshot.patch`로 즉시 백업(커밋 금지 정책과 양립하는 보험). 이번 사고에서 이것이 있었으면 복구가 즉시 완결됐다.
