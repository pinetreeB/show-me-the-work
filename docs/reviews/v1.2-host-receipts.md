## 1. Claude Code — 완전한 block→recovery→cap 사이클 (우하 pane 라이브, 2026-07-12 새벽)

유도 시나리오: `tmp/cc-receipt/demo.py` 생성(print 한 줄) → 의도적으로 검증 없이 턴 종료 → 관측.

1. **change 기록**: `PostToolUse:Write says: fable-lite 원장: 변경 1건 기록 / recorded 1 change(s).`
2. **1차 Stop block**: `Stop hook error: fable-lite Stop gate: 변경 파일이 있지만 성공한 검증 증거가 없습니다…` — 미검증 완료 선언이 실제로 차단됨.
3. **회복 시도**: 워커가 `cd … && python tmp/cc-receipt/demo.py; echo "exit=$?"` 실행 → 출력 `receipt-demo`, exit=0 관측.
4. **2차 Stop block**: 성공 실행이었으나 원장에 성공 검증으로 인정되지 않음 → 재차단.
   - 원인(워커 자체 분석 + 좌상 동의): 이 환경의 Bash 훅 payload에 구조화 종료코드가 없고, 출력(`receipt-demo`)에 성공 토큰이 없어 텍스트 폴백이 보수적 False — **false-negative(과차단) 방향의 마찰**. agy F1(false-red)과 증상은 같은 계열이나 경로가 다름.
5. **3차 allow (fail-open cap)**: `Stop says: fable-lite Stop gate allow.` — MAX_STOP_BLOCKS=2 캡이 설계대로 자기치유(영구 락아웃 없음).

**판정**: 게이트 차단·회복 강제·fail-open 캡 모두 실호스트에서 설계대로 동작.
**파생 백로그**: 성공 토큰 없는 커스텀 스크립트 출력(echo 래핑 포함)의 검증 인식 — P1-5 corpus 계열 (보수 방향이라 안전하나 마찰).

## 2. Codex CLI — 세션 상시 발동 (우상 pane, 2026-07-11~12 연속 관측)

- 세션 부팅·매 프롬프트: `UserPromptSubmit hook (completed) — hook context: fable-lite 활성화: 작업 규율을 절차로 적용하세요. mode=deep`
- 도구 실행마다: `Running PostToolUse hook: fable-lite post-tool ledger`
- 턴 종료마다: `Stop hook (completed) — warning: fable-lite Stop gate allow.`
- block/recovery 경로: 실 Codex 세션 유도는 이번에 생략 — P4 수정분 `tests/test_p4_codex_install_regressions.py`가 **외부 cwd + 격리 HOME에서 설치된 훅 4종의 실행과 미검증 변경 Stop=block**을 서브프로세스로 관측(F10 수용기준).

**판정**: repo cwd 라이브 발동 확증 + 외부 cwd는 격리 실행으로 block까지 관측.
