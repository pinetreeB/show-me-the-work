## 1. 07-20 실사고 원인 재분석 및 잔여 문제 실재 여부

### 원인 재분석: 워커의 산출물이 왜 "타 에이전트 파기"로 차단되었는가?
R2 게이트는 워커가 자신의 산출물을 직접 쓰는 행위 자체는 허용합니다(`_owned_by_unsettled_peer`가 자신일 경우 통과). 07-20 실측에서 차단이 발생한 정확한 구조적 원인은 다음과 같습니다.

1. **오케스트레이터의 파일 선점 (Placeholder)**: 오케스트레이터(Claude)가 워커에게 작업을 위임하기 전에 셸이나 도구를 사용해 산출물 대상 파일(예: `tmp/b1-design.md`)을 미리 생성(touch)했을 확률이 높습니다. 이 경우 파일의 소유권은 오케스트레이터에게 귀속되며, 오케스트레이터가 대기 중이므로 해당 턴은 "미정산(unsettled)" 상태가 됩니다.
2. **워커의 셸 명령 쓰기**: 워커(agy)가 해당 파일에 리포트를 쓰기 위해 편집 툴(`write_to_file`) 대신 셸 명령(`run_command` 상의 `cat <<EOF > ...` 등)을 사용했습니다.
3. **R2 차단 정합성**: 셸 명령을 가로챈 R2 게이트는 타겟 파일이 "타 에이전트(오케스트레이터)의 미정산 파일"임을 확인하고, 안전을 위해(`peer_unsettled_revision`) 정상적으로 fail-closed 차단했습니다.

### 잔여 문제 실재 여부 (3개 PR 머지 후)
**판정: 여전히 재현됩니다 (문제 실재함).**
- **B4 (Goals 네임스페이스)**: 목표(goals) 파일 충돌을 해결했을 뿐 임의 산출물 파일 경로(`tmp/`)와는 무관합니다.
- **PR D (CODEX identity)**: Codex 어댑터의 런타임 식별자를 보정하는 패치이므로, Antigravity CLI(`agy`)의 워커 간 소유권 충돌과는 무관합니다.
- **PR E (NEW-02)**: 동적 셸 명령(`parse_unable_*`)에 대한 오라벨링 개선입니다. 이 건은 파싱 실패가 아니라, 타겟이 정적으로 완벽히 파악(`resolved`)된 후 "소유권 방어" 로직에 의해 차단된 것이므로 무관합니다.

따라서 멀티에이전트 환경에서 오케스트레이터가 파일을 생성하고 워커가 셸 명령으로 덮어쓰려 할 때 산출물이 증발하는 UX 결함은 그대로 남아있습니다.

## 2. 설계 질문에 대한 답변 및 권고안

### 목표 경계
R2의 Fail-closed 원칙(위임받지 않은 무단 파기 차단)을 절대로 약화해서는 안 됩니다. 즉, 차단을 '허용'으로 바꾸는 (b)안은 기각해야 합니다. 차단은 유지하되 증발하는 UX만 구제해야 합니다.

### 대안 평가
- **(c) Deny 시 파일 내용을 stderr/메시지로 반환해 오케스트레이터가 재생성**: 
  - **기각**. 차단된 셸 명령이나 방대한 파일 내용 전체를 에러 메시지에 담아 워커에게 반환하면 컨텍스트 윈도우(토큰)를 심각하게 낭비합니다. 워커가 이 메시지를 그대로 복사해 오케스트레이터에게 전달해야 하므로 매우 비효율적입니다.
  
- **(a) 정산 대기 유예 (Pending Quarantine)**:
  - **채택**. R2가 파괴적 명령을 차단(`deny`)할 때, 툴 페이로드의 명령 문자열(또는 툴 입력값)을 `.fable-lite/quarantine/blocked-<timestamp>-<agent_key>.sh` 등의 덤프 파일로 로컬 디스크에 백업합니다.

### 최종 설계 권고 (Quarantine 방안)

1. **격리소 백업**: `oma_hook.py` (또는 `adapter_observation.py`의 R2 핸들러)에서 차단 판정이 내려지면, 차단된 명령(수천 줄의 `cat` 산출물 포함)을 `quarantine/` 디렉토리에 파일로 씁니다.
2. **UX 메시지 개선**: R2 게이트 차단 사유(reason) 문자열에 다음을 추가합니다.
   > *"작성하려던 파일 내용은 유실되지 않았으며 `.fable-lite/quarantine/blocked-...` 에 안전하게 보관되었습니다. 오케스트레이터에게 해당 스크립트를 실행해 반영해달라고 요청하세요."*
3. **오케스트레이터 회수**: 파일 소유권자인 오케스트레이터는 워커의 보고를 받은 후, 격리소에 있는 스크립트를 스스로 실행(본인 소유이므로 R2 통과)하여 안전하고 매끄럽게 산출물을 병합할 수 있습니다.

이 설계는 R2의 강력한 방어선을 1px도 훼손하지 않으면서, 에이전트가 소모한 컴퓨팅/토큰(산출물 텍스트) 자원 소실을 완벽하게 방어하는 최적의 우회로를 제공합니다.
