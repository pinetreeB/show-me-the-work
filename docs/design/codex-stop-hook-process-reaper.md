# 제안: Codex stop 훅 기반 자기 세션 프로세스 정리 (미구현 — 맥락·의견 기록)

> 2026-07-12. 상태: **아이디어 기록만, 구현 착수 금지** (사용자 지시). v1.2 P1 진행과 무관한 별도 트랙.
> 발의: 사용자(이승현) — "fable-lite 훅이 codex에도 적용되고 작업 종료 시 뭔가를 강제하는 훅이니까,
> codex에게 업무 종료 시 자신이 스폰한 서브에이전트만 건드려서 종료하게 하는 방식은 어떤가."

## 1. 문제 맥락 (2026-07-12 실측)

Codex multi-agent(codex-cli 0.144.1 + omo 4.16.3)는 **서브에이전트 스레드마다 stdio MCP
풀세트(~8종: context7·codegraph·lsp-daemon·git-bash-mcp·chrome-devtools·wmux bundle 등)
+ node_repl.exe를 새로 스폰하고, 스레드 종료 후 회수하지 않는다.**

- 실측: node.exe 934개 + node_repl.exe 94개 = **48.5GB** (하루 반 축적분).
  시간대 분포로 "세션 재시작 문제"가 아니라 **작업 중 상시 축적**임을 확정 (22시에만 219개 생성).
- 대비: Claude Code 서브에이전트는 동일 프로세스 내 대화 스레드(연산=클라우드, MCP=부모 공유)라
  같은 기간 자식 node 11개. 아키텍처 차이가 원인이며 codex 쪽 설정으로는 못 막는다.
- 현행 방어선: `C:\Users\rotat\scripts\mcp-reaper.ps1` v2 + Task Scheduler 매시 정각.
  Tier1(고아 30분+) / Tier2(2h+ 유휴 AND CPU<60s), 살아있는 claude/agy/wmux 소속 제외,
  살아있는 codex 세션의 최초 도구 세트(첫 자식+5분 창) 보호. 실측 48.5→14.2GB 무손상 회수.
  상세: 메모리 위키 `reference/tools/reference_codex_mcp_zombie_fix.md`.

## 2. 발의 아이디어에 대한 판정

**방향은 옳다. 단, 귀속 단위를 한 단계 조정해야 안전하다.**

- **불가능한 부분**: "자신이 스폰한 서브에이전트*만*" 식별. 병렬 스레드 15개가 낳은 MCP
  프로세스들은 전부 같은 codex.exe 부모 밑에 동일 명령줄로 붙는다. 프로세스 테이블에
  스레드 귀속 표식(env/cmdline 마커)이 없어, 스레드 단위 정리는 시간창 휴리스틱에 의존하게
  되고 → 옆에서 아직 실행 중인 스레드의 서버를 죽일 위험이 있다. subagent_stop 훅 단위
  정리는 이 이유로 **기각**.
- **가능한 수정안**: 귀속 단위를 **턴 종료(stop) + 자기 세션(부모 PID) 스코프**로 변경.
  stop 시점엔 해당 세션의 서브에이전트가 전부 종료된 상태이므로 "세션 최초 도구 세트를
  제외한 자기 PID 소속 전부 = 시체"라는 판정이 결정적(deterministic)이다. 다른 pane의
  codex/claude/agy는 부모 PID가 다르므로 구조적으로 건드릴 수 없다.
- fable-lite는 codex stop 훅 인프라를 이미 보유(v1.2 P0 codex self-locating으로 타 프로젝트
  발동 경로도 정비 중). 훅 발동 시 자기 codex PID는 부모 프로세스 체인으로 획득 가능
  (self-locating과 동일 계열 문제).

## 3. 권장 구도 (구현 시)

- **훅 = 1차 방어선**: stop 발동 즉시 자기 세션 잔재 회수 → 축적 자체를 발생 지점에서 차단.
- **시간당 리퍼 = 2차 백스톱 유지**: 훅은 배신한다 — agy 1.1.1 훅 엔진 미발동 6회 실측
  전례(`docs/reviews/p9-agy-live-hooks.md`). codex 비정상 종료(OOM·강제킬)나 훅 미발동 시
  스케줄러가 회수. 백스톱 제거 금지.
- **fail-open 필수**: 정리 실패가 codex 턴 완료를 막으면 안 된다. 로그만 남기고 통과.
- **성능 무손실 원칙**: 죽이는 것은 "종료된 스레드의 잔재"뿐. 스레드 동시 수·MCP 종류·effort
  등 능력 관련 설정 불변 (사용자 Premium-Max 정책).

## 4. 스코프 경계 (로드맵 합의와의 긴장)

v-next 로드맵의 "하지 않기로 3자 합의"에 **zero-dep·플랫폼 중립 core 유지**가 있다.
Windows 프로세스 테이블 순회·Stop-Process는 명백히 플랫폼 종속이므로:

- core(verify_state 등)에 넣지 않는다. **호스트별 옵션 애드온**(예: `contrib/` 또는
  `.codex/hooks` 확장 스크립트, Windows 전용 명시)으로만 구현한다.
- 게이트 로직과 무관한 위생(hygiene) 훅이므로 v1.2 하드게이트 대상도 아니다.
- 위치 제안: v1.2 P1 이후 별도 소품(트랙 외) 또는 v2.0 어댑터 계약 논의에 부속.
