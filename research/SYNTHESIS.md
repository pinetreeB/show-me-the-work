# 리서치 종합 — Fable 재현 시도들 & Fable vs Opus 근본 차이

> 원본: `codex-fable-vs-opus.md` · `agy-prior-attempts.md` · `opus-fablize-analysis.md`

## 1. 결론 (한 문단)

Fable 5는 Opus 튜닝판이 아니라 **가중치부터 다른 상위 티어(Mythos-class)** 모델이며, 격차는 "쉬운 일 전반"이 아니라 **길고 복잡한 hard tail**(장기 자율 코딩·스펙 밖 결함 발견·자발적 함의 추적)에 집중된다. 그러나 3개 독립 소스(fablize 통제실험, Tessl 917시나리오 평가, 시스템 카드)가 수렴하는 실증은: **닫힌 과제에선 Opus도 사실상 동률이고, 좋은 하네스/스킬의 효과(+17점)가 모델 교체 효과(+0.9점)보다 크다**. 따라서 fable-lite의 승부처는 모델 흉내가 아니라 **"절차의 강제(하드 게이트)"** — fablize가 검증한 방향이 옳고, 그 빈틈(준수 검증·자동 트리거·범위 통제·한국어·멀티에이전트)을 메우는 것이 개선 기회다.

## 2. 기존 재현 시도 계보

| 시도 | 접근 | 실효성 | 한계 |
|------|------|--------|------|
| **fablize** (fivetaku, 사용자 설치본 2.1.0) | A/B 19런+실작업 26세션(~1,500 tool calls) 통제 비교 → **전이 검증된 절차만** 훅/팩/상태파일로 강제 | 검증 접지·완주 게이트·조사 프로토콜·조기종료 차단이 측정 가능하게 개선 | 능력 상한은 못 올림(자인). 소프트 지시 우회 가능, 준수 검증 부재 |
| **Ralph Loop + Opus** (XDA 기고) | 에이전트 루프 반복 + 마일스톤 분할 + CLAUDE.md | 장시간 자율작업 체감 **~80% 회복** 주장 | 루틴 작업엔 효과 미미, 토큰 소모 급증, 정량 비교 없음 |
| **프롬프트 기법 일반** (Self-Refine·CoVe·Self-Consistency·메타프롬프팅) | 멀티패스로 얕은 추론을 시간/토큰으로 보완 | 검증·일관성 향상 보고 다수 | 단일 턴 능력 자체는 불변 |
| **하네스 설계 계보** (SWE-agent ACI·Aider·OpenHands) | 모델 친화적 인터페이스 + 단계 게이트 + 분해 | 하네스만으로 벤치 대폭 상승 선례 다수 | 모델별 재조정 필요 |
| **Distillation** (합성데이터→파인튜닝) | 교사 추론 궤적을 가중치에 각인 | 체급 돌파는 유일하게 가능 | ⚠️ **Fable 5는 distillation 시도를 classifier로 차단** — 정책·안전상 fable-lite 범위 밖 |

## 3. Fable vs Opus — 근본 차이 (공식 근거)

- **정체**: Fable 5 = Mythos 5와 동일 가중치 + 안전장치(cyber/bio/distillation classifier, 감지 시 Opus 4.8 fallback·refusal). Mythos-class는 Opus 위 티어. (발표문·시스템 카드)
- **벤치 격차는 hard tail 집중**: SWE-bench Pro 80.0 vs 69.2 / FrontierCode Diamond 29.3 vs 13.4(2배+) / SWE-bench Verified 95.0 vs 88.6 / Terminal-Bench 84.3 vs 82.7(근소). 과제가 길수록 격차 확대(공식 문구).
- **제품 차이**: always-on adaptive thinking(끌 수 없음), effort 체계, 2배 가격($10/$50 vs $5/$25), 30일 보존. 1M 컨텍스트·128k 출력은 Opus 4.8도 동일 — 차이는 크기가 아니라 **오래 안 무너지는 활용력**.
- **반례·주의**: 실사용에서 "대부분 과제는 Opus로 충분"(Reddit 다수), 코드리뷰 정밀도는 오히려 Opus 우위(CodeRabbit), Fable도 하네스 없인 장기 자율성 불안정.

## 4. 전이 가능 vs 불가 (3소스 수렴 — fable-lite 설계 기준표)

| 전이 **가능** (절차 — 우리가 만들 것) | 전이 **불가** (가중치 — 정직하게 에스컬레이션) |
|---|---|
| 검증 접지 (만들면 실제로 돌려보고 관측) | 스펙 밖 결함의 자발적 발견 (주입 실험으로 반증됨) |
| 분해 + 증거 게이트 완주 (goals.py류) | 열린 과제의 창의 디테일 |
| 체계적 조사 (재현→3+ 가설→인과사슬) | 자발적 전파 깊이 ("함의 한 걸음 더") |
| 조기종료 방지 (미이행 약속 차단 훅) | hard-tail 추론 상한 (FrontierCode류) |
| 증거 기반 진행 보고 (fabricated status 차단) | 비전/문서/과학 원시 인지 |
| fresh-context verifier·병렬 서브에이전트 | 적은 프롬프트로 의도를 읽는 능력 |
| 메모리 파일·스킬·컨텍스트 엔지니어링 | medium effort로 상위 effort를 넘는 효율 |

## 5. fablize 2.1.0의 빈틈 = fable-lite 개선 기회 (우하 Opus 1인칭 평가)

핵심 통찰: **하드 게이트(결정론적 차단)는 확실히 먹히고, 소프트 지시는 모델이 합리화로 우회한다.** fablize는 하드(D)와 소프트(A·B) 사이 중간 강도가 없다.

- **H1 팩 준수 검증**: "3+ 가설 세워라"를 실제 세웠는지 확인하는 게이트 없음
- **H2 goals.py 자동 트리거**: 소프트 지시뿐이라 우회 쉬움
- **H3 범위 이탈 감지**: "scope 내 유지"가 지시문뿐, 탐지 수단 없음
- **H4 한국어 라우팅**: router.sh 영어 키워드 전용
- **H5 멀티에이전트·플랫폼 독립**: Claude Code+bash 전용 → wmux 4-pane(Opus·Gemini·Codex)에서 못 씀
- 기타: 증분 커밋 규율(M1)·자기 diff 리뷰(M2)·에스컬레이션 프로토콜(M4)·측정 인프라 분리(L1)·Windows 네이티브(L5)
